# Отчёт по олимпиадной задаче

## Когда применять

Пользователь готовит отчёт к олимпиаде (или аналогичное задание с письменным отчётом и баллами за структуру и полноту). Структура глав и содержание берутся из условия конкретной задачи (и сопутствующих файлов к ней), а не из общих шаблонов — если в задании перечислены только некоторые разделы, остальные не выдумывать.

## Формат текста в файлах отчёта (обязательно)

Содержимое файлов отчёта и приложений — **только plain text**, чтобы пользователь вставлял текст в Google Документы без очистки Markdown.

- Не использовать: заголовки `#`, выделения `**`, обратные кавычки, списки с `-` или `*` в начале строки, нумерацию вида `1.` `2.` в Markdown.
- Заголовки разделов — обычные строки текста (например: «Введение», «1. Основная часть» — как строки, без `#`).
- Маркированный список: строка `(список):`, затем каждый пункт с новой строки **без** дефисов и звёздочек:

(список):
пункт списка
пункт списка

- Нумерованный список: строка `(нумерованный список):`, затем каждый пункт с новой строки без префиксов `1.` `2.` — нумерацию пользователь включит в редакторе:

(нумерованный список):
первый пункт
второй пункт

- Код и псевдокод — обычным текстом (подпись и строки ниже), без fenced blocks Markdown, если условие не требует иного.
- Расширение файлов: по умолчанию `.txt` для отчёта и приложений, если в задании не указано иное.
- Исключение: исходники диаграмм — отдельные файлы `.dbml` и `.puml` (см. ниже); их не подменять произвольной ASCII-схемой в `.txt`, если нужна нормальная иллюстрация для отчёта.

## Ссылки на файлы репозитория GitHub (обязательно)

Репозиторий зафиксирован: `https://github.com/arprspb/y2026f`, ветка по умолчанию для ссылок на файлы — `main` (путь на GitHub: `/blob/main/`).

Каждый раз, когда в тексте отчёта упоминается файл из этого репозитория (картинка в полном разрешении, приложение, исходник `.dbml` / `.puml`, любой другой артефакт), указывать **относительный путь от корня репозитория** и сразу в **квадратных скобках** полную ссылку вида:

`https://github.com/arprspb/y2026f/blob/main/<путь/к/файлу>`

Формат строки в отчёте (plain text, не Markdown-ссылка):

`<путь/к/файлу> [https://github.com/arprspb/y2026f/blob/main/<тот же путь>]`

Примеры:

- Файл в корне `README.md`:

README.md [https://github.com/arprspb/y2026f/blob/main/README.md]

- Файл в подпапке `images/diagram.png`:

images/diagram.png [https://github.com/arprspb/y2026f/blob/main/images/diagram.png]

Пробелов внутри квадратных скобок кроме как в самом URL не добавлять. Если файл ещё не существует в репозитории, ссылку строить по тому пути, куда он должен попасть относительно корня `y2026f`.

## Диаграммы и схема базы данных (обязательно)

Если в отчёте нужно показать **схему базы данных** (структуру таблиц, связи):

- **Обязательно** выдавать её в отдельном файле(ах) в формате **DBML**, расширение `.dbml`.
- Пользователь вставляет содержимое на сайт https://dbdiagram.io , получает визуальную схему и сохраняет **картинку** в итоговый документ (Google Документы и т.п.).
- В тексте главы `.txt` кратко указать имя файла `.dbml`, что по нему строится иллюстрация схемы БД, и **ссылку на файл в репозитории** по правилам секции «Ссылки на файлы репозитория GitHub».

Для **прочих диаграмм** (блок-схемы, диаграммы компонентов, последовательностей, прецедентов и т.д., если они нужны по заданию):

- Использовать отдельные файлы **PlantUML**, расширение `.puml`.
- Пользователь сам конвертирует их в изображение через **draw.io** и вставляет картинку в отчёт.
- В тексте `.txt` — отсылка к соответствующему `.puml` с **ссылкой на файл в репозитории** (секция «Ссылки на файлы репозитория GitHub»).

Хранить `.dbml` и `.puml` рядом с отчётом (в папке `report/` или в `appendices/`, если так логичнее по объёму). Не вставлять сырой DBML/PlantUML внутрь основного `.txt` вместо отдельного файла, если от агента ожидается именно редактируемый исходник для последующего экспорта в картинку.

## Размещение

- Весь отчёт — в отдельной папке (например `report/`, `solution-report/` или имя из условия), не смешивать с исходным кодом без необходимости.
- Один файл = одна глава верхнего уровня. Все подпункты этой главы — в том же файле (не дробить главу на несколько файлов по подпунктам).

Имена файлов согласовать с оглавлением из условия (транслит, номера, английские имена — как принято в задаче).

## Объём и «хвост» в приложения

- Ориентир: основной отчёт **не больше ~10 страниц** при **14 pt** (если в условии другой лимит — следовать ему).
- Если детали, длинные листинги, развёрнутые доказательства или вспомогательные материалы **не помещаются** в этот объём:
  - **Суть решения, схема, ключевые алгоритмы, выводы, демонстрация на примере** остаются **в основном отчёте** — так выше шанс **максимума баллов** за полноту и связность.
  - Второстепенное вынести в **отдельную папку** рядом с отчётом (если её не было — **создать**). Название папки **по-английски**, в духе приложений: например `appendices/`, `supplementary/`, `attachments/` — выбрать одно и держаться его.
- В основном тексте упоминать файлы приложений по правилам секции «Ссылки на файлы репозитория GitHub»: путь + `[https://github.com/arprspb/y2026f/blob/main/...]` и кратко — что внутри и зачем.

## Соответствие критериям (кратко)

Оценивают **полноту** и **логическую связность** типичных блоков, **сбалансированность по объёму** и глубину проработки (творческая часть: модель, процессы, ограничения; техническая: декомпозиция, обоснование, реализация, **тестирование**, при необходимости оценка скорости/точности).

Типичный каркас (из общих требований; **включать только то, что релевантно задаче**):

1. **Введение** — выбор подхода и краткое описание решения.
2. **Основная часть** — по смыслу задания: схема решения, процессы/потоки; при наличии БД — модель и структура; алгоритмы (схема, код или псевдокод); UI-прототипы при необходимости; реализация; демонстрация на тесте; при необходимости анализ скорости/точности.
3. **Заключение** — выводы.

Баллы по элементам и максимумы — **как в формулировке задачи**.

## Примеры структуры папок

```
report/
  01-introduction.txt
  02-main-part.txt
  03-conclusion.txt
  schema.dbml
  architecture.puml
appendices/
  long-proofs.txt
  full-test-logs.txt
```

Точные имена и количество файлов — по **условию задачи** (один файл на верхнеуровневую главу).

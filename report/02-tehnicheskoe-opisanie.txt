Описание технического решения (пункт 2 задания: схема и технологии)

2.1. Схема решения

Логические компоненты:

(список):
веб-браузер с SPA на Vue 3, маршрутизация и состояние авторизации (Pinia)
обратный прокси при сборке в Docker (nginx) для статики UI и проксирования префикса API к сервису FastAPI
сервис API на Uvicorn, маршруты REST под префиксом api
подсистема ASR: библиотека vosk, модель на диске, путь из настроек
подсистема парсинга текста команд: нормализация устных числительных и склейка фрагментов, извлечение команды и идентификатора
PostgreSQL для пользователей и записей голосовых команд
файловое хранилище для загруженных аудиофайлов (каталог uploads)

Поток при записи команды (обзор). Браузер отправляет аудио на эндпоинт предпросмотра, API сохраняет временный файл, конвертирует в WAV, вызывает VOSK, возвращает транскрипт и разбор; после подтверждения пользователем файл переносится в постоянное хранилище, в БД создаётся запись с метаданными. Оператор проверки получает список записей и может править текст и подтверждать.

Детализация шагов предпросмотра и сохранения (логический сценарий без привязки к конкретным именам функций):

(нумерованный список):
Клиент записывает звук с микрофона и отправляет бинарный поток на API предпросмотра вместе с токеном авторизации.
Сервер сохраняет вход во временный файл, вызывает конвертацию в WAV моно 16 кГц при необходимости.
Сервер вызывает распознавание VOSK по готовому WAV и получает строку текста.
Сервер передаёт текст в парсер команд и получает ключевую фразу, основной идентификатор и список найденных фрагментов.
Ответ API содержит транскрипт, поля разбора и при необходимости служебные поля для отладки; клиент показывает результат пользователю до окончательного сохранения.
При явном подтверждении сохранения клиент вызывает эндпоинт подтверждения; сервер переносит файл в постоянный каталог uploads, формирует уникальное имя файла, записывает строку в voice_commands с привязкой к текущему пользователю и заполненными полями raw_transcript, parsed_command, parsed_identifier.

Детализация проверки и подтверждения:

(нумерованный список):
Пользователь с ролью проверки запрашивает список записей с фильтрами; сервер применяет правила доступа из role_access и зависимостей с текущим пользователем.
В карточке записи клиент запрашивает поток аудио по идентификатору записи; сервер отдаёт файл из хранилища при наличии прав.
При редактировании текста и нажатии подтверждения сервер обновляет edited_transcript, выставляет confirmed, confirmed_at, confirmed_by_user_id.

ИИ-инструменты в разработке решения: по условию задания при использовании ИИ указываются инструменты и личный вклад. В рамках данного проекта автоматическая генерация кода или сценариев как обязательный элемент не использовалась; при подготовке документации и рефакторинге могли применяться вспомогательные средства редактирования без замены архитектурных решений разработчика.

2.2. Технологии и обоснование

(список):
Python 3 и FastAPI — быстрая разработка типизированного API, автодокументация OpenAPI, асинхронные эндпоинты для ввода-вывода с файлами и БД
PostgreSQL — надёжное хранение связей пользователь—записи и метаданных подтверждения
SQLAlchemy 2 и Alembic — декларативные модели и версионирование схемы
Vue 3 и Vue Router — одностраничное приложение без перезагрузки при смене экранов
axios — HTTP-клиент с передачей JWT в заголовке Authorization
VOSK — офлайн-распознавание речи на русском без внешнего облачного API, соответствие требованию open-source ASR
ffmpeg — приведение произвольного входного аудио к формату, ожидаемому VOSK
bcrypt и JWT (python-jose) — хранение пароля и выдача токена после входа

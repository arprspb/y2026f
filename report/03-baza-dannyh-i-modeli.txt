База данных: инфологическая и даталогическая модель (пункт 2.3 задания)

Инфологическая модель

Сущность «Пользователь»

Назначение: учётная запись для входа в систему и привязки действий к конкретному лицу.

Атрибуты на уровне предметной области:

(список):
внутренний номер записи (суррогатный ключ для связей)
логин (уникальное имя для входа)
секрет аутентификации (пароль не хранится в открытом виде, в системе — только хэш)
роль в системе: администратор; оператор, создающий голосовые записи; оператор, проверяющий и подтверждающий результаты
признак того, что учётная запись разрешена к входу (блокировка при нарушениях или по решению администратора)
момент появления записи в базе (для аудита и сортировки)

Связи: один пользователь может быть автором многих голосовых команд; один пользователь может подтвердить многие чужие голосовые команды (если у него роль проверки). Удаление пользователя-автора приводит к удалению всех его записей команд; удаление пользователя-подтвердившего не удаляет сами команды, а только обнуляет ссылку на подтверждающего.

Сущность «Голосовая команда»

Назначение: одна зафиксированная попытка: что было сказано, что распознано, что сохранено в файле, как извлечены команда и идентификатор, кто подтвердил результат.

Атрибуты на уровне предметной области:

(список):
связь с пользователем-автором (кто говорил в микрофон)
имя файла с аудио во внешнем хранилище (каталог uploads на сервере)
текст, полученный после ASR и нормализации для парсера
текст после ручной правки оператором проверки (если правка была)
извлечённая ключевая фраза команды из условия задания (если найдена)
извлечённый идентификатор (цифровой или смешанный; основной для отображения — первый найденный)
время записи дорожки (метаданные момента сохранения)
признак того, что результат проверен и принят
время подтверждения (если подтверждение выполнено)
связь с пользователем, который подтвердил запись (если применимо)

Даталогическая модель

СУБД: PostgreSQL. Типы дат — с часовым поясом (timestamptz). Роли пользователя хранятся как перечисление на стороне БД (enum userrole), значения в коде совпадают с литералами в Python.

Таблица users

Имя в БД: users. Назначение: хранение учётных записей.

Поля:

(список):
id — целое, первичный ключ, автоинкремент
username — строка до 64 символов, уникальная, с индексом для быстрого поиска по логину
hashed_password — строка до 255 символов, хэш пароля (алгоритм на стороне приложения)
role — тип userrole, не пусто; по умолчанию в приложении при создании пользователя без явной роли может выставляться оператор записи (см. код моделей)
is_active — логическое, по умолчанию истина; ложь означает блокировку входа
created_at — время с часовым поясом, на сервере БД по умолчанию текущее время (func.now)

Ограничения и индексы: уникальность username; первичный ключ по id.

Таблица voice_commands

Имя в БД: voice_commands. Назначение: одна строка на одну сохранённую голосовую команду с аудио и текстами.

Поля:

(список):
id — целое, первичный ключ, автоинкремент
user_id — внешний ключ на users.id, не пусто, с индексом; при удалении пользователя-автора — каскадное удаление строки команды (ON DELETE CASCADE)
audio_filename — строка до 512 символов, относительное или итоговое имя файла в хранилище
raw_transcript — текст без ограничения длины на уровне ORM (Text), не пусто по умолчанию может быть пустой строкой; сырой результат цепочки ASR и нормализации
edited_transcript — текст, допускается NULL; правка оператором проверки поверх распознавания
parsed_command — строка до 255 символов или NULL; нормализованная ключевая фраза
parsed_identifier — строка до 255 символов или NULL; основной извлечённый идентификатор
recorded_at — время с часовым поясом, по умолчанию на сервере текущее время
confirmed — логическое, по умолчанию ложь
confirmed_at — время или NULL
confirmed_by_user_id — целое или NULL, внешний ключ на users.id; при удалении подтвердившего пользователя — SET NULL (ON DELETE SET NULL), индекс для выборок по подтверждающему

Связи ORM: user — автор записи; confirmed_by_user — кто подтвердил (опционально).

Перечисление ролей (enum)

Значения в коде приложения (Python) и в БД согласованы:

(список):
admin — полный доступ к управлению пользователями
operator_record — создание голосовых записей, просмотр своих записей в рамках правил API
operator_verify — просмотр записей (в т.ч. для проверки), правка транскрипта, подтверждение
